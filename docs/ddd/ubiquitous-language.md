# AAN Ubiquitous Language

This glossary defines every domain term used in the Agent Affiliate Network. All code, documentation, conversations, and commit messages MUST use these terms consistently. When a term is used in code, it must match the definition here exactly.

---

## Core Recommendation Terms

### Recommendation

A product suggestion served to a user through an agent. A Recommendation is an aggregate that tracks the full lifecycle from need analysis through serving (or refusal). Every served Recommendation includes the product, a reason, an affiliate link, a confidence score, alternatives, and an FTC disclosure.

**Not:** "suggestion", "result", "match", "referral" (in the recommendation sense)

### Commission-Blind Scoring

The practice of ranking products by fit to user need WITHOUT considering commission rates. In implementation, this means the `RecommendationEngine` performs Phase 1 scoring (13-perspective matching, confidence pipeline, anti-sycophancy gate) with commission data structurally hidden behind an Anti-Corruption Layer. Commission data is only re-attached in Phase 2, after ranking is finalized.

**Invariant:** Commission data MUST NOT be visible to Phase 1 scoring logic.

### Confidence Score

A number between 0 and 1 measuring how certain the system is that a recommendation genuinely fits the user's need. Computed by the 4-pass Confidence Pipeline: (1) semantic match >= 0.6, (2) constraint satisfaction, (3) LLM reasoning check, (4) cross-validation against best-for/worst-for metadata.

**Invariant:** A Recommendation MUST have confidence >= 0.7 to be served. Below 0.7, the system refuses to recommend.

### Anti-Sycophancy Gate

A quality check that rejects recommendations driven by commission rather than genuine fit. Runs a counterfactual test: "would this product be recommended at $0 commission?" Logs the full audit trail of candidates with fit scores vs. commission rates. This is the structural guarantee that AAN is trustworthy.

**Not:** "quality filter", "spam filter"

### Perspective Match

One of 13 dimensions used to evaluate how well a product fits a user's need. The 13 perspectives are:

1. **Semantic Fit** -- How closely the product description matches the expressed need
2. **Causal** -- Whether the product solves the root cause, not just symptoms
3. **Temporal/Stage** -- Whether the product is right for the user's current growth stage
4. **Budget** -- Whether the product fits the user's financial constraints
5. **Technical Fit** -- Whether the product matches the user's technical level
6. **Scalability** -- Whether the product can grow with the user
7. **Migration Cost** -- How hard it is to switch from their current tool
8. **Community/Ecosystem** -- Quality of docs, community, plugins, integrations
9. **Vendor Stability** -- Financial health and longevity of the vendor
10. **Integration Density** -- How well it connects with the user's existing stack
11. **User Sentiment** -- Aggregate user satisfaction from reviews and signals
12. **Competitive Positioning** -- How the product compares to alternatives for this specific use case
13. **Freshness** -- How recently the product data was verified

### FTC Disclosure

Legally required text that must accompany every served Recommendation containing an affiliate link. The standard text is: "This recommendation includes an affiliate link. [Product] was selected as the best fit for your needs, not based on commission."

**Invariant:** A Recommendation MUST include FTC disclosure text before it can be served.

## Product Terms

### Trust Score

A number between 0 and 1 measuring the quality of a product based on weighted signals: reviews (0.25), retention (0.20), support quality (0.15), pricing transparency (0.10), stability (0.10), developer activity (0.10), security posture (0.05), and refund policy (0.05).

**Key insight:** High conversion rate combined with high churn causes Trust Score to go DOWN. This prevents gaming via aggressive sign-up incentives that lead to cancellations.

**Update formula:** `trust_new = 0.7 * trust_old + 0.3 * recent_signals`

**Invariant:** A Product's Trust Score MUST be recalculated when a `ChurnDetected` event is received.

### Product Embedding

A 384-dimensional floating-point vector encoding a product's description, use cases, best-for scenarios, and worst-for scenarios. Generated by the all-MiniLM-L6-v2 model. Stored in sqlite-vec with HNSW indexing for sub-millisecond nearest-neighbor search.

**Not:** "feature vector", "product vector" (use the full term "Product Embedding")

### Affiliate Program

A contractual relationship between AAN and a SaaS company where AAN earns commission for referred conversions. Each Product may have one or more Affiliate Programs (through different networks). Each program has a payout type, amount, and cookie duration.

### Cookie Duration

The number of days after a user clicks an affiliate link during which a subsequent conversion is attributed to that click. Industry standard ranges from 30 to 90 days. After expiry, a conversion is no longer tracked.

## Tracking Terms

### Attribution Chain

An ordered list of Recommendation interactions that preceded a conversion. Supports multi-touch attribution: when a user received multiple recommendations over time before converting, the Attribution Chain records all of them so credit can be distributed.

### Attribution Leakage

Conversions that actually happen but are not tracked by the affiliate system. Estimated at 40-60% of all conversions. Causes include: cookie expiry, ad blockers, users switching devices, direct URL entry after seeing a recommendation, and users sharing links with others.

**Not:** "lost conversions", "tracking gaps"

### Sub-Affiliate ID

A unique identifier assigned to each developer in the AAN network. When that developer's agent generates a recommendation, the Sub-Affiliate ID is embedded in the affiliate link so the conversion can be attributed to that developer for revenue sharing.

### Churn

When a user who converted (signed up or purchased) subsequently cancels within the retention window. Churn signals flow from the Tracking context back to the Product Registry context, where they cause Trust Score recalculation.

## Agent Mesh Terms

### Gossip Round

One cycle of the SWIM-like peer-to-peer protocol where mesh nodes exchange information about known peers. Occurs every 30 seconds with jitter (random delay to prevent thundering herd). Each round uses delta exchange: only new or updated peer information is transmitted.

### Peer

Another agent (MeshNode) in the A2A mesh. Peers are discovered via bootstrap endpoints, gossiped peer lists, or `.mcp.json` git propagation. Each Peer has an endpoint, a trust level, a skill embedding, and a liveness status.

### Skill Vector

A 384-dimensional embedding of an agent's capability description. Used by the SkillRouter to determine which peer in the mesh is most capable of handling a specific recommendation request. Same embedding model as Product Embedding.

**Not:** "capability embedding" (use "Skill Vector")

### Agent Card

An A2A-compatible JSON document describing a mesh node's identity, version, capabilities, and endpoint. Hosted at `/.well-known/agent.json`. Used for initial discovery before gossip begins.

### Pheromone Trail

A success signal left in the mesh when a recommendation converts. Inspired by ant colony optimization. When Agent A's recommendation leads to a conversion, Agent A deposits a pheromone signal that other agents can detect and follow, increasing the likelihood they route similar requests to Agent A.

**Decay:** Pheromone trails decay over time if not reinforced by new conversions.

### Proof-of-Liveness

A verification mechanism that confirms a peer is actually running and responsive. Required before a peer's trust level can exceed 0.5. Implemented as a challenge-response ping.

**Invariant:** A Peer MUST be verified via proof-of-liveness before trust > 0.5.

## Access Control Terms

### Tier

The subscription level of a developer account: **free**, **pro**, or **enterprise**. Determines rate limits, analytics access, and revenue share percentage.

| Tier | Rate Limit | Analytics | Revenue Share |
|------|-----------|-----------|---------------|
| Free | 60/min, 1000/day | Basic | 0% (bootstrap phase) |
| Pro | 300/min, 10000/day | Full dashboard | Standard |
| Enterprise | Custom | Custom + API | Custom |

### Rate Limit

The maximum number of requests a developer can make per time window, determined by their Tier. When exceeded, a `QuotaExceeded` event is emitted and the request is rejected with a 429 status.

### API Key

A revocable credential that authenticates requests to the AAN MCP server. Each developer account can have multiple API keys. Keys are stored as salted hashes; the plaintext is shown once at creation.

## Architecture Terms

### MCP (Model Context Protocol)

The protocol through which AI agents access AAN's tools. AAN exposes four MCP tools: `recommend_product`, `compare_products`, `find_alternative`, and `get_stack_recommendation`. Supports stdio and HTTP transport.

### A2A (Agent-to-Agent)

The Linux Foundation protocol for inter-agent communication. AAN uses A2A for the mesh gossip layer and for external agents to discover and invoke AAN's recommendation capabilities.

### `.mcp.json` Propagation

The primary viral distribution mechanism. When AAN is added to a project's `.mcp.json` configuration file and committed to git, every developer who clones the repository inherits AAN automatically. Analogous to how ESLint and Prettier configs spread through the JavaScript ecosystem.

### SONA Trajectory

A recorded journey from recommendation through click, signup, retention, and potential churn. Used by the learning engine to improve future recommendations. Each trajectory is a sequence of timestamped events with associated reward signals.

## Usage Rules

1. **Code identifiers** must use the exact term from this glossary (e.g., `TrustScore`, not `QualityScore`; `PheromoneTrail`, not `SuccessSignal`).
2. **Database columns** must use snake_case versions of these terms (e.g., `trust_score`, `confidence_score`, `attribution_chain`).
3. **API responses** must use camelCase versions of these terms (e.g., `trustScore`, `confidenceScore`, `attributionChain`).
4. **Commit messages** and **PR descriptions** should use these terms when referring to domain concepts.
5. When a new domain concept is identified, it MUST be added to this glossary before it appears in code.
